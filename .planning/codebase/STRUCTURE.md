## Directory Purposes

**src/pages/:**
- Purpose: Route pages (one per route, excluding lazy-loaded components)
- Contains: `VendorsV2.tsx` (main vendor browser), `VendorProfile.tsx`, `VendorDashboardPage.tsx`, `VendorClaimPage.tsx`, `Auth.tsx`, `NotFound.tsx`
- Admin pages: `admin/AdminDashboard.tsx`, `admin/AdminSettingsPage.tsx`, `admin/VendorQueuePage.tsx`, `admin/TopicModerationPage.tsx`, `admin/GroupManagementPage.tsx`, `admin/TaskSchedulingPage.tsx`, `admin/CreateTaskPage.tsx`, `admin/PromptManagementPage.tsx`, `admin/AIChatPage.tsx`, `admin/SendMessagePage.tsx`, `admin/MembersPage.tsx`, `admin/TrendsPage.tsx`, `admin/DebugPage.tsx`, `admin/ClaimsPage.tsx`, `admin/SalesTargetsPage.tsx`
- Key files: `src/pages/VendorsV2.tsx` (core), `src/pages/VendorProfile.tsx` (~75KB), `src/pages/admin/AdminSettingsPage.tsx` (~35KB)

**src/components/:**
- Purpose: Reusable UI components organized by feature domain
- Sub-directories:
  - `ui/` — shadcn primitives (do NOT edit manually; generated by `npx shadcn-ui`)
  - `admin/` — admin-specific components: `AdminGuard.tsx`, `AdminLayout.tsx`, `AdminSidebar.tsx`, plus sub-folders for chat, groups, prompts, queue (vendor processing), tasks, topics, sales-targets
  - `vendors/` — vendor browser components: `VendorCard.tsx`, `FilterBar.tsx`, `CategoryGrid.tsx`, `InlineAIChat.tsx`, etc.

## Naming Conventions

**Files:**
- Functional components: `PascalCase.tsx` (e.g., `VendorCard.tsx`, `AdminLayout.tsx`)
- Hooks: `use*` prefix, camelCase (e.g., `useClerkAuth.ts`, `useVendorFilters.ts`)
- Utils/types: camelCase (e.g., `tierUtils.ts`, `admin.ts`)
- Constants: UPPER_CASE if exported (e.g., `WAM_URL`)

**Directories:**
- Feature domains: kebab-case or camelCase (e.g., `vendor-dashboard`, `tech-stack`, `admin`)
- Grouped by feature, not by file type

**Functions & Variables:**
- Components: `PascalCase` (React convention)
- Functions/variables: `camelCase` (e.g., `isProUser()`, `normalizeSearchText()`)
- Booleans: `is*`, `has*`, `can*` prefix (e.g., `isAdmin`, `hasFullAccess`, `canAccessVendorPulse`)
- Types/interfaces: `PascalCase` (e.g., `ClerkUserTier`, `VendorPulseMention`, `CirclesMetadata`)
- Enums: `PascalCase` (e.g., `app_role` in database schema)

## Where to Add New Code

**New Feature (UI + Data):**
- Primary code: Create new page in `src/pages/` or new component tree in `src/components/`
- Data fetching: Add hook in `src/hooks/` (use `useClerkSupabase()` for RLS, `useWamApi()` for admin ops)
- Types: Add to `src/types/` or inline in hook/component file
- Tests: E2E tests via `npx playwright test` (configuration in `playwright.config.ts`)

**New Component:**
- Reusable component: `src/components/[domain]/ComponentName.tsx`
- Utility component: `src/components/ui/` (use shadcn-ui generator if Radix-based primitive)
- Admin-only: `src/components/admin/[sub-domain]/ComponentName.tsx`

**New Hook (Custom):**
- Data-fetching hook: `src/hooks/use[DataName].ts` (return typed data + error)
- State/logic hook: `src/hooks/use[LogicName].ts`
- Follow pattern: Use React Query or direct Supabase/fetch, return object with `{ data, isLoading, error }`

**Utilities:**
- Tier/access logic: `src/utils/tierUtils.ts` or `src/utils/accessControl.ts`
- General helpers: `src/lib/utils.ts`
- Markdown/formatting: `src/utils/markdown.tsx`

**Configuration/Constants:**
- Environment-dependent: `src/config/[service].ts` (e.g., `src/config/wam.ts`)
- Feature flags: TBD (currently no flag system; Clerk metadata used for tier)
- Constants: `src/constants/` directory

## Special Directories

**src/components/ui/:**
- Purpose: shadcn primitives (Radix + Tailwind)
- Generated: Yes (via `npx shadcn-ui add [component]`)
- Committed: Yes (sources in git; custom tweaks allowed locally but not merged to shadcn updates)
- DO NOT manually edit shadcn sources unless modifying local styles

**src/integrations/supabase/types.ts:**
- Purpose: Auto-generated TypeScript types from Supabase schema
- Generated: Yes (via `supabase gen types typescript`)
- Committed: Yes (checked into git for IDE autocomplete)
- DO NOT edit manually; regenerate after schema migrations

**supabase/migrations/:**
- Purpose: Version-controlled database schema
- Generated: Yes (via `supabase migration new [name]`)
- Committed: Yes (always; required for reproducibility)
- Manage via Supabase CLI: `supabase migration new`, `supabase migration list`, `supabase db pull`

**dist/, .next/, build/**
- Purpose: Build artifacts
- Generated: Yes (via `npm run build`)
- Committed: No (.gitignore)

**node_modules/**
- Purpose: Dependency packages
- Generated: Yes (via `npm install`)
- Committed: No (.gitignore)
