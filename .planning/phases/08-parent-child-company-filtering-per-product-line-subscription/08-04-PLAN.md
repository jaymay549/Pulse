<objective>
Thread the active product line slug from `useActiveProductLine()` context through ALL vendor dashboard components that call RPCs. Update their React Query cache keys to include the slug, ensuring data refreshes when the vendor switches product lines.

Purpose: Per D-11 and D-13, all dashboard data must be scoped to the selected product line. Without this, switching product lines in the dropdown shows stale data from a different product line.
Output: All dashboard components pass slug to RPCs and include slug in query keys.
</objective>

For EACH of the listed dashboard component files, apply this pattern:

**Step 1: Add import at top of file:**
```typescript
import { useActiveProductLine } from "@/hooks/useActiveProductLine";
```

**Step 2: Get the active slug inside the component function body** (near other hook calls):
```typescript
const { activeProductLine } = useActiveProductLine();
const productLineSlug = activeProductLine?.slug ?? null;
```

**Step 3: Update ALL React Query `queryKey` arrays** to include `productLineSlug` at the end. This is CRITICAL per Pitfall 2 — without this, switching product lines will show stale cached data.

Examples of what to change in each file:

**DashboardOverview.tsx:**
- `["vendor-recent-mentions", vendorName]` -> `["vendor-recent-mentions", vendorName, productLineSlug]`
- `["vendor-sentiment-history", vendorName]` -> `["vendor-sentiment-history", vendorName, productLineSlug]`
- Pass `productLineSlug` to `fetchVendorPulseFeed({ ..., productLineSlug })` if calling that function
- Pass `productLineSlug` to any RPC call as `p_product_line_slug` parameter

**DashboardMentions.tsx:**
- `["vendor-respond-mentions", vendorName]` -> `["vendor-respond-mentions", vendorName, productLineSlug]`
- `["vendor-respond-responses", vendorName]` -> `["vendor-respond-responses", vendorName, productLineSlug]`
- Pass `productLineSlug` to `fetchVendorPulseFeed` calls

**DashboardIntel.tsx:**
- `["intel-own-profile", vendorName]` -> `["intel-own-profile", vendorName, productLineSlug]`
- `["vendor-trend", vendorName]` -> `["vendor-trend", vendorName, productLineSlug]`
- `["intel-competitors", vendorName]` -> `["intel-competitors", vendorName, productLineSlug]`
- Pass `productLineSlug` to `fetchVendorProfile`, `fetchVendorTrend`, `fetchComparedVendors` calls

**DashboardDimensions.tsx:**
- `["dashboard-dimensions", vendorName]` -> `["dashboard-dimensions", vendorName, productLineSlug]`
- `["vendor-entity-id", vendorName]` -> `["vendor-entity-id", vendorName]` (entity ID doesn't change by product line — keep as is)
- `["dashboard-dimension-mentions", vendorName, ...]` -> `["dashboard-dimension-mentions", vendorName, productLineSlug, ...]`

**DashboardCategories.tsx:**
- `["vendor-categories", vendorName]` -> `["vendor-categories", vendorName, productLineSlug]`

**DashboardDealerSignals.tsx:**

**Step 4: Pass slug to underlying fetch functions.** For components that call functions from `useSupabaseVendorData.ts` (like `fetchVendorPulseFeed`, `fetchVendorProfile`), pass `productLineSlug` as the `productLineSlug` parameter. These functions already accept and forward it to the RPCs.

For components that call RPCs directly (like `supabase.rpc("some_rpc", { ... })`), add `p_product_line_slug: productLineSlug` to the parameters IF the RPC accepts that parameter. If the RPC does not accept a product line slug parameter, just update the queryKey (the RPC will filter by vendor name only, which is fine for RPCs that aren't product-line-aware yet).

**IMPORTANT:** Do NOT change component prop signatures. The slug comes from the `useActiveProductLine()` context hook, not from props. This avoids prop-drilling (per RESEARCH.md anti-pattern).
