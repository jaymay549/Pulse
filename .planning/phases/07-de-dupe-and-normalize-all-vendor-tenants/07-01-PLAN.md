Create a SQL migration file at `supabase/migrations/20260416100000_vendor_dedup_audit_rpc.sql` with:

1. **`admin_audit_vendor_duplicates()`** RPC (SECURITY DEFINER, plpgsql, STABLE):
   Returns a JSONB object with three arrays:

   a) `"duplicate_groups"` -- Groups of vendor_mentions that share the same `lower(trim(vendor_name))` but have different `vendor_entity_id` values (including NULL). For each group:
      - `normalized_name`: the lowercased trimmed vendor_name
      - `variants`: array of objects `{ vendor_name, vendor_entity_id, canonical_name (from joined vendor_entities or null), mention_count }`
      - `total_mentions`: sum of all mention counts in the group
   
   b) `"unlinked_vendors"` -- Distinct vendor names from vendor_mentions where `vendor_entity_id IS NULL`, with mention_count, ordered by mention_count DESC. Each entry: `{ vendor_name, mention_count }`.

   c) `"orphan_metadata"` -- Rows from vendor_metadata where `lower(vendor_name)` does not match any `lower(canonical_name)` in vendor_entities AND does not match any `lower(alias_text)` in vendor_alias_mappings. Each entry: `{ vendor_name, has_logo (boolean), has_description (boolean) }`.

   d) `"orphan_profiles"` -- Rows from vendor_profiles where `lower(vendor_name)` does not match any `lower(canonical_name)` in vendor_entities AND does not match any `lower(alias_text)` in vendor_alias_mappings. Each entry: `{ vendor_name, user_id, is_approved }`.

   e) `"stats"` -- Summary object: `{ total_mentions, linked_mentions, unlinked_mentions, total_entities, total_aliases, duplicate_group_count }`.

2. Grant: `GRANT EXECUTE ON FUNCTION public.admin_audit_vendor_duplicates() TO authenticated, service_role;`

The RPC is read-only (STABLE). It scans vendor_mentions, vendor_entities, vendor_alias_mappings, vendor_metadata, and vendor_profiles to produce a comprehensive audit.

Use CTEs for each section to keep the query readable. The duplicate detection logic:
```sql
WITH name_groups AS (
  SELECT lower(trim(vendor_name)) AS norm_name,
         vendor_entity_id,
         vendor_name,
         COUNT(*) AS mention_count
  FROM public.vendor_mentions
  WHERE is_hidden = false
  GROUP BY lower(trim(vendor_name)), vendor_entity_id, vendor_name
),
multi_entity_names AS (
  SELECT norm_name
  FROM name_groups
  GROUP BY norm_name
  HAVING COUNT(DISTINCT COALESCE(vendor_entity_id::text, 'NULL')) > 1
     OR COUNT(DISTINCT vendor_name) > 1
)
```
This catches both "same name pointing to different entities" and "same normalized name with different casing/spacing".

**Step 1: Create a reusable name normalization function:**
```sql
CREATE OR REPLACE FUNCTION public._normalize_vendor_name(p_name TEXT)
RETURNS TEXT
LANGUAGE sql
IMMUTABLE
AS $$
  SELECT initcap(trim(regexp_replace(coalesce(p_name, ''), '\s+', ' ', 'g')));
$$;
```
This trims whitespace, collapses multiple spaces to one, and applies title case (initcap). Grant to authenticated, anon, service_role.

**Step 2: Normalize vendor_entities.canonical_name:**
```sql
UPDATE public.vendor_entities
SET canonical_name = public._normalize_vendor_name(canonical_name),
    slug = public._slugify_vendor_token(public._normalize_vendor_name(canonical_name)),
    updated_at = now()
WHERE canonical_name IS DISTINCT FROM public._normalize_vendor_name(canonical_name);
```

**Step 3: Normalize vendor_alias_mappings.alias_text** (always lowercase trimmed):
```sql
UPDATE public.vendor_alias_mappings
SET alias_text = lower(trim(regexp_replace(alias_text, '\s+', ' ', 'g')))
WHERE alias_text IS DISTINCT FROM lower(trim(regexp_replace(alias_text, '\s+', ' ', 'g')));
```

**Step 4: Normalize vendor_mentions.vendor_name:**
```sql
UPDATE public.vendor_mentions
SET vendor_name = public._normalize_vendor_name(vendor_name)
WHERE vendor_name IS DISTINCT FROM public._normalize_vendor_name(vendor_name);
```

**Step 5: Normalize vendor_metadata.vendor_name:**
```sql
UPDATE public.vendor_metadata
SET vendor_name = public._normalize_vendor_name(vendor_name)
WHERE vendor_name IS DISTINCT FROM public._normalize_vendor_name(vendor_name);
```

**Step 6: Normalize vendor_profiles.vendor_name:**
```sql
UPDATE public.vendor_profiles
SET vendor_name = public._normalize_vendor_name(vendor_name),
    updated_at = now()
WHERE vendor_name IS DISTINCT FROM public._normalize_vendor_name(vendor_name);
```

**Step 7: Re-register canonical names as aliases** (in case normalization changed them):
```sql
INSERT INTO public.vendor_alias_mappings (alias_text, vendor_entity_id, confidence, source)
SELECT lower(ve.canonical_name), ve.id, 1.0, 'normalization'
FROM public.vendor_entities ve
WHERE NOT EXISTS (
  SELECT 1 FROM public.vendor_alias_mappings am
  WHERE am.alias_text = lower(ve.canonical_name)
    AND am.vendor_entity_id = ve.id
)
ON CONFLICT (alias_text) DO UPDATE
  SET vendor_entity_id = EXCLUDED.vendor_entity_id,
      confidence = GREATEST(vendor_alias_mappings.confidence, EXCLUDED.confidence);
```

Each UPDATE must use `IS DISTINCT FROM` to avoid unnecessary writes. Add a comment header explaining the migration purpose.

## Trust Boundaries

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Integrity | _normalize_vendor_name | mitigate | initcap() may produce wrong casing for acronyms (e.g., "CDK" -> "Cdk"); however this matches the existing pattern in admin_merge_vendors which uses user-provided canonical names. The normalization function only handles whitespace/casing baseline -- acronym-specific canonical names are set via admin_merge_vendors RPC. |
| T-07-02 | Integrity | vendor_entities unique constraint | mitigate | Normalization could create duplicate canonical_name values after initcap. The UPDATE uses ON CONFLICT-safe patterns -- if two entities normalize to the same name, the second UPDATE will fail on the UNIQUE constraint. Plan 02 handles merging these duplicates. Run audit RPC first to identify conflicts before applying normalization. |
| T-07-03 | Availability | Large UPDATE on vendor_mentions | accept | vendor_mentions may have 100K+ rows. UPDATE with IS DISTINCT FROM is efficient (only touches changed rows). Supabase handles this within normal migration timeouts. |
| T-07-04 | Integrity | alias_text UNIQUE constraint | mitigate | Normalizing alias_text could create duplicates. Use `ON CONFLICT` or `WHERE alias_text IS DISTINCT FROM` to handle gracefully. If two aliases normalize to the same text, keep the one with higher confidence. |
</threat_model>
