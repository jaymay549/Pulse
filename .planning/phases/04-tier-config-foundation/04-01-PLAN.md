<task type="auto">
  <name>Task 1: Create tier_component_config migration with seed data and upsert RPC</name>
  <files>supabase/migrations/20260415000000_tier_component_config.sql</files>
  <action>
Create a Supabase migration file that:

1. **Creates the `tier_component_config` table** in the `public` schema:
   - `id` UUID PRIMARY KEY DEFAULT gen_random_uuid()
   - `tier` TEXT NOT NULL — one of 'tier_1', 'tier_2', 'test' (no unverified — unverified vendors don't see the dashboard)
   - `component_key` TEXT NOT NULL — matches DashboardSection values: 'intelligence', 'overview', 'segments', 'mentions', 'profile', 'intel', 'dimensions', 'demo-requests', 'screenshots', 'categories', 'dealer-signals'
   - `visibility` TEXT NOT NULL DEFAULT 'full' CHECK (visibility IN ('full', 'gated', 'hidden'))
   - `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()
   - UNIQUE constraint on (tier, component_key) — one row per tier-component pair

2. **Do NOT enable RLS** on this table — per PROJECT.md "Out of Scope": "RLS policy changes for tier_component_config — frontend-only gating layer on top of existing RLS safety net". Add a COMMENT explaining this is intentional.

3. **Seed default rows** for all 3 tiers x 11 components = 33 rows:
   - **tier_1**: 'full' for intelligence, overview, segments, profile, intel, screenshots, categories. 'hidden' for mentions, dimensions, dealer-signals, demo-requests (matching current T2_ONLY_SECTIONS hardcoding).
   - **tier_2**: 'full' for ALL 11 components.
   - **test**: Copy tier_2 defaults (all 'full') — admin can customize later per ACONF-03.

4. **Create `upsert_tier_component_config` RPC function** (SECURITY INVOKER, not DEFINER):
   - Parameters: `p_tier TEXT`, `p_component_key TEXT`, `p_visibility TEXT`
   - Body: INSERT INTO tier_component_config (tier, component_key, visibility, updated_at) VALUES (p_tier, p_component_key, p_visibility, now()) ON CONFLICT (tier, component_key) DO UPDATE SET visibility = EXCLUDED.visibility, updated_at = now();
   - Returns the upserted row.
   - This RPC is called by the admin UI to persist individual cell changes immediately (per ACONF-04).

5. **Create `get_tier_component_config` RPC function** (SECURITY INVOKER):
   - No parameters.
   - Returns all rows from tier_component_config ordered by tier, component_key.
   - Simple SELECT * wrapper to keep data access through RPCs for consistency.

Add a table COMMENT: 'Admin-configurable tier-component visibility. No RLS — read by admin panel only. Phase 4 (v1.1).'
