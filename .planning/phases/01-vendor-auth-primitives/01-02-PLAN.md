Create `src/pages/VendorLoginPage.tsx` implementing a two-step OTP login flow.

The page has two steps rendered inside a single Card. Step 1 shows the email input; step 2 replaces step 1 content with the OTP input. Use `useState` for step management (not routing).

**Step 1 — Email entry:**
- Card title: "Vendor Portal" (h2, `text-xl font-black tracking-tight`)
- Card description: "Enter your email to receive a one-time code."
- Label "Email address" with `htmlFor="vendor-email"` / `id="vendor-email"` pair
- Input `type="email"` with `placeholder="you@company.com"`
- Button "Send Code" full-width, variant="default", `disabled` while sending
- While sending: button label changes to "Sending..." with `Loader2` spinner (`animate-spin h-4 w-4 mr-2`, `aria-hidden="true"`)
- On `signInWithOtp` success: transition to step 2, show `toast.success("Code sent! Check your email.")`
- On error: display error message below input using `<p role="alert" className="text-destructive text-sm mt-2">`
  - If error indicates unknown email (any non-rate-limit error from signInWithOtp with shouldCreateUser: false): "This email isn't registered. Contact your sales representative to get access."
  - If error message contains "rate" or "60 seconds" (case-insensitive): "Too many attempts. Please wait a minute before requesting another code."

**Step 2 — OTP verification:**
- Confirmation text: "We sent a 6-digit code to {email}. Enter it below." (`text-sm text-muted-foreground`)
- `InputOTP` component with `maxLength={6}`, `value={otp}`, `onChange={setOtp}`
  - Wrap slots in `InputOTPGroup`, render 6 `InputOTPSlot` components with `index={0..5}`
  - Add `font-mono text-xl font-black` class to InputOTP wrapper for monospace digits
  - Minimum slot height: `h-11` (44px touch target)
- Button "Verify Code" full-width, variant="default", `disabled` while verifying or `otp.length < 6`
- While verifying: button label "Verifying..." with `Loader2` spinner
- On `verifyOtp` success (data.session is non-null): `navigate('/vendor-dashboard')` (no toast needed, navigation is confirmation)
- On `verifyOtp` error: show `<p role="alert" className="text-destructive text-sm mt-2">` with "That code is expired or invalid. Request a new code below."
  - On error, add `border-destructive` class to InputOTPGroup wrapper
- Ghost button "Request a new code" — calls `signInWithOtp` again with same email (resets otp state, shows toast on success)
- Ghost button "Use a different email" — resets to step 1 (clears email and otp state)

**Session expired banner (AUTH-08 companion):**

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User email input -> Supabase Auth API | Untrusted email submitted to signInWithOtp |
| OTP code input -> Supabase Auth API | 6-digit code submitted to verifyOtp |
| Session token -> localStorage | JWT stored under vendor-auth key |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Spoofing | VendorLoginPage signInWithOtp | mitigate | shouldCreateUser: false prevents self-registration; only admin-provisioned emails receive OTP (AUTH-09) |
| T-01-06 | Tampering | VendorLoginPage OTP brute force | accept | Supabase enforces rate limiting on OTP requests (1/60s) and verification attempts by default; SEC-02 (custom rate limiting) is v2 |
| T-01-07 | Information Disclosure | VendorLoginPage error messages | mitigate | Error messages for unknown email say "Contact your sales rep" — do not reveal whether email exists in auth.users specifically; generic enough to avoid enumeration |
| T-01-08 | Denial of Service | Navigation vendor login link | accept | Public link to login page; no auth required to view; standard web pattern |
</threat_model>
