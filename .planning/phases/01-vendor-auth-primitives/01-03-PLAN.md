Create `src/components/vendor-auth/VendorAuthGuard.tsx` modeled after AdminGuard:

```typescript
import { Navigate } from "react-router-dom";
import { Loader2 } from "lucide-react";
import { useClerkAuth } from "@/hooks/useClerkAuth";
import { useVendorSupabaseAuth } from "@/hooks/useVendorSupabaseAuth";

interface VendorAuthGuardProps {
  children: React.ReactNode;
}

const VendorAuthGuard = ({ children }: VendorAuthGuardProps) => {
  const { isLoading: clerkLoading, isAdmin } = useClerkAuth();
  const { isLoading: vendorLoading, isAuthenticated: isVendorAuth } = useVendorSupabaseAuth();

  // Show loading while either auth system is initializing
  if (clerkLoading || vendorLoading) {
    return (
      <div className="min-h-screen flex items-center justify-center bg-background">
        <Loader2 className="h-8 w-8 animate-spin text-muted-foreground" aria-label="Loading..." />
      </div>
    );
  }

  // AUTH-05: Admin bypass — Clerk admin can always access vendor dashboard
  if (isAdmin) return <>{children}</>;

  // AUTH-04: Vendor auth — valid vendor Supabase session passes
  if (isVendorAuth) return <>{children}</>;

  // Neither: redirect to vendor login
  return <Navigate to="/vendor-login" replace />;
};

export default VendorAuthGuard;
```

Key details:
- Loading state uses `bg-background` and `text-muted-foreground` (matches UI-SPEC, adapts to dark mode — differs from AdminGuard's hardcoded `bg-zinc-950`)
- Loader2 has `aria-label="Loading..."` for screen reader accessibility (per UI-SPEC)
- Clerk admin check comes FIRST — if isAdmin is true, bypass vendor auth entirely (AUTH-05)
- Vendor Supabase auth checked second — if isVendorAuth is true, allow access (AUTH-04)
- If neither, redirect to `/vendor-login` (not `/vendors` like AdminGuard)
- Default export (matches AdminGuard pattern for lazy loading in App.tsx)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| VendorAuthGuard | Decides whether to allow or redirect based on two auth sources |
| VendorDashboardPage vendor_logins query | Maps auth.uid() to vendor_name; if compromised, could show wrong vendor's data |
| App.tsx route registration | Public route /vendor-login must not require auth; /vendor-dashboard must require guard |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Elevation of Privilege | VendorAuthGuard | mitigate | Guard checks isAdmin (Clerk) OR isVendorAuth (Supabase) — neither auth system's tokens are interchangeable; Clerk JWT cannot produce a vendor Supabase session |
| T-01-10 | Spoofing | VendorDashboardPage vendor_logins lookup | mitigate | Query uses vendorSupabase client (with vendor JWT); RLS policy ensures auth.uid() = user_id; vendor can only resolve their own vendor_name |
| T-01-11 | Information Disclosure | VendorDashboardPage vendor data | accept | Phase 1 does not add RLS to vendor data tables (vendor_profiles, vendor_mentions, etc.) — that is Phase 3. Vendor sees same data as before via vendorName prop. Risk is low because vendor_logins.vendor_name correctly scopes the view. |
| T-01-12 | Tampering | App.tsx /vendor-login route | accept | Public route; no auth needed to view login page; standard pattern |
</threat_model>
