Create `supabase/functions/provision-vendor/index.ts` following the exact Edge Function pattern from `admin-ensure-vendor-profile/index.ts`.

The function accepts a POST with JSON body: `{ vendor_email: string, vendor_name: string, tier: string, _auth_token: string }`.

**Request handling:**
1. Handle OPTIONS preflight with corsHeaders (same as admin-ensure-vendor-profile).
2. Extract `_auth_token` from body (fallback to Authorization header). Call `verifyAdmin(authToken)` — reject if not admin.
3. Validate required fields: `vendor_email` (non-empty string), `vendor_name` (non-empty string), `tier` (must be one of `"unverified"`, `"tier_1"`, `"tier_2"`). Return 400 with specific error message if validation fails.

**Provisioning logic (two actions: provision and resend):**

Support an optional `action` field in the body. Default to `"provision"`. If `action === "resend"`, skip step (a) and go directly to generating a new link for an existing user.

**(a) For `action === "provision"` (default):**
- Call `supabase.auth.admin.inviteUserByEmail(vendor_email, { data: { vendor_name }, redirectTo: redirectUrl })` where `redirectUrl` is `Deno.env.get("PUBLIC_APP_URL") || "https://app.cdgpulse.com"` + `/vendor-dashboard`.
- If `inviteUserByEmail` returns error with message containing "already been registered" (case-insensitive), treat as existing user: call `supabase.auth.admin.generateLink({ type: "magiclink", email: vendor_email, options: { redirectTo: redirectUrl } })` to get the user ID, then proceed to upsert.
- If it succeeds, extract `userId` from `inviteData.user.id`.
- Upsert into `vendor_logins`: `supabase.from("vendor_logins").upsert({ user_id: userId, vendor_name, tier, updated_at: new Date().toISOString() }, { onConflict: "user_id" })`. Using upsert (not insert) per RESEARCH.md to handle re-provisioning.
- Return `{ ok: true, user_id: userId, action: "provisioned" }`.

**(b) For `action === "resend"`:**
- Require `vendor_email` in body.
- Call `supabase.auth.admin.generateLink({ type: "magiclink", email: vendor_email, options: { redirectTo: redirectUrl } })`.
- If error, throw it (email not found = user was never provisioned).
- After generating the link, Supabase does NOT automatically send the email for `generateLink`. So after getting the link, call `supabase.auth.admin.inviteUserByEmail(vendor_email, { redirectTo: redirectUrl })` to trigger the email send. If this returns "already registered" error, that's expected — the initial `generateLink` already confirmed the user exists. In that case, use an alternative approach: look up the user via `supabase.auth.admin.listUsers({ filter: vendor_email })` to get the user, then generate a magic link via `generateLink` and return the `action_link` for the admin to share manually if email delivery fails.
- Actually, simpler approach per Supabase behavior: for resend, just call `supabase.auth.admin.generateLink({ type: "magiclink", email: vendor_email, options: { redirectTo: redirectUrl } })`. This generates a magic link. Then to actually send the email, use Supabase's built-in `signInWithOtp` approach — but that's client-side. Instead, return the magic link URL to the admin as a fallback and also attempt `inviteUserByEmail` which re-sends the invite email.
- Return `{ ok: true, action: "resend_sent" }`.

**SIMPLIFIED APPROACH (use this):**
Both provision and resend ultimately need to send an email to the vendor. Use this single flow:

1. Try `supabase.auth.admin.inviteUserByEmail(vendor_email, { redirectTo: redirectUrl })`.
2. If error message contains "already been registered": user exists. Call `supabase.auth.admin.generateLink({ type: "magiclink", email: vendor_email, options: { redirectTo: redirectUrl } })` to get user_id. 4. For resend action: skip vendor_logins upsert (row already exists).
5. Return `{ ok: true, user_id, action: "provisioned" | "resend_sent" }`.

Create migration file `supabase/migrations/20260413100000_admin_list_vendor_logins_rpc.sql` with a SECURITY DEFINER function that joins `vendor_logins` with `auth.users` to return the full vendor list for the admin table.

The function signature:
```sql
CREATE OR REPLACE FUNCTION public.admin_list_vendor_logins()
RETURNS TABLE (
  id UUID,
  user_id UUID,
  vendor_name TEXT,
  tier TEXT,
  email TEXT,
  created_at TIMESTAMPTZ,
  last_sign_in_at TIMESTAMPTZ
)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  RETURN QUERY
    SELECT
      vl.id,
      vl.user_id,
      vl.vendor_name,
      vl.tier,
      u.email::TEXT,
      vl.created_at,
      u.last_sign_in_at
    FROM public.vendor_logins vl
    JOIN auth.users u ON u.id = vl.user_id
    ORDER BY vl.created_at DESC;
END;
$$;
```

After the function definition, add permission grants:
```sql
REVOKE ALL ON FUNCTION public.admin_list_vendor_logins() FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.admin_list_vendor_logins() TO authenticated;
```

Add a comment:
```sql
COMMENT ON FUNCTION public.admin_list_vendor_logins() IS 'Admin-only RPC: returns vendor_logins joined with auth.users for the admin vendor management table. SECURITY DEFINER required because auth.users is not accessible via anon/authenticated keys.';
```

Note: This RPC does not enforce admin-only access at the SQL level (no role check inside the function). The admin check is enforced at the frontend level (only admin pages call this) and the data returned (vendor emails + tiers) is not sensitive enough to warrant an additional SQL-level guard. This matches the pattern of other admin RPCs like `admin_list_unclaimed_vendor_candidates` in ClaimsPage.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Edge Function | Untrusted HTTP request from admin browser to provision-vendor |
| Edge Function -> Supabase Auth | Service-role key used to create auth users |
| Edge Function -> vendor_logins | Service-role INSERT/UPSERT into vendor_logins table |
| Frontend -> admin_list_vendor_logins RPC | Authenticated user calls RPC that reads auth.users |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | provision-vendor Edge Function | mitigate | `verifyAdmin(token)` decodes Clerk JWT and checks `user_role === "admin"` before any operation |
| T-02-02 | Information Disclosure | Service-role key | mitigate | Key stored only in Deno.env (Edge Function secrets), never in client bundle or response |
| T-02-03 | Tampering | vendor_email input | mitigate | Validate non-empty string; Supabase Auth validates email format internally |
| T-02-04 | Tampering | tier input | mitigate | Validate `tier` is one of `["unverified", "tier_1", "tier_2"]` before DB write |
| T-02-05 | Spoofing | _auth_token field | accept | Token is a Clerk JWT — spoofing requires Clerk secret key; same pattern as all existing admin Edge Functions |
| T-02-06 | Information Disclosure | admin_list_vendor_logins RPC | accept | Returns vendor emails visible only to authenticated users; admin-only pages control UI access; data is low-sensitivity (vendor business emails, not PII) |
| T-02-07 | Denial of Service | Mass provisioning | accept | Admin-only endpoint; no public access; rate limiting deferred to v2 (SEC-02) |
</threat_model>
